# 계획 대 계획 판정: matter 문서 패키지(Plan A) vs 리포 RP/CP 체계(Plan B)

작성일: 2026-06-11
방법: 8개 차원 × (독립 평가 → 적대적 재검증) + 비교 프레임 자체의 누락·편향 비판. 총 17개 독립 분석 트랙, 모든 근거는 파일 직접 확인 기반. 평가 인용 근거 중 부정확한 것은 검증 단계에서 적발·정정함(본문에 반영).

## 1. 판정 결과표

| 차원 | Plan A | Plan B | 승자 | 검증 결과 |
|---|---|---|---|---|
| 범위 완전성 | 6 | 7.5 | **B** | 유지 (high) |

## 2. 왜 B가 이기는가 — 한 문장

**Plan A는 "강제 장치가 없는 헌장"이고 Plan B는 "구멍이 있으나 실제로 폭주를 막아본 기계"다.** 거의 모든 차원에서 같은 패턴이 반복됐다: A는 개념·어휘·의미론이 더 넓고 정확하지만 전부 산문 선언(v0.1 표)이고, B는 커서→팩→게이트→영수증으로 이어지는 기계 집행 구조가 실재하며 320+팩 실행으로 가동이 입증된다.

핵심 증거:
- **착수 가능 단위**: A의 TUW 204개는 전건 동일 보일러플레이트 의존성·완료조건의 제목 목록(문서 18 스스로 "상세 확장 필요" 자인)이며, ID 체계가 문서 17·18·24 사이에서 서로 충돌하고 자기 형식(4세그먼트)마저 위반 — **착수 가능 인스턴스 0개**. B는 next-pack-queue 커서, 리스크 기반 팩 사이징, unit별 status 추적이 기계가독으로 완결.
- **게이트 강제력**: A의 CG-001~008은 의미론적으로 더 강하지만(런타임 테스트 요구) 검증 스크립트·증빙 양식·작업 단위 바인딩이 0건이라 "주장하면 통과". B는 436줄 검증기 + read-only 이종 모델 리뷰 영수증(위조 출력 5종 무효화 하드닝) + 무효 시도 영구 보존 + 진도 단조성 검사. 실제 BLOCK 사례(C-LDIP-01, P0 3건 + 프롬프트 인젝션 벡터 적발)까지 보유.
- **적응성**: A는 Feedback/Loop edge·Learning Ledger 등 어휘는 더 넓지만 계획 재생성·개정 절차·일탈 기록 장치가 전무 — 자기 "확정 전제"가 현실과 충돌해도 감지 못함(Sprint 0 착수 지시가 이미 닫힌 RP00~06과 중복). B는 live-state-overrides-plan 재동기화, override_reason 규율, 계획 밖 트랙(LDIP)을 표준 절차로 흡수한 완결 사례 보유.
- **리스크**: 식별의 구체성(명시적 Risk Register 12건)은 A 우위지만, 무력화에서 B 압도 — sensitivity-first 분류가 569팩에 실인스턴스화, 권한·감사 커널 최선행 배치가 실행으로 입증, 12개 외부 표준(OWASP/NIST/PIPA 등) 매핑 보유(A는 0건).

## 3. A가 이기거나 앞서는 것 (B에 이식할 가치)

1. **아키텍처·기술 구체성 (A 승)**: 문서 08의 M365 명세(Office.js+MSAL/NAA, Smart Alerts send mode 3종, Graph 3대 ID, 조직 배포)가 Microsoft Learn 공식 문서와 교차검증 전 항목 일치 — 2025년 레거시 토큰 차단 이후의 공식 방향을 정확히 반영. 반면 B의 RP08/RP22/email-dms 계약에는 플랫폼 기술 결정이 0건이고, 스택은 계약 "candidate" vs NARR-013 vs 실제 구현(node:http/Vite)의 3중 불일치. 스토리지 소재(SharePoint/OneDrive 원본)도 A만 결정함.
2. **리스크 식별 레지스터**: RISK-001~012 + OQ-001~015 (소유자·우선순위 포함) — B에는 통합 제품 리스크 레지스터가 없음(계약별 분산).
3. **런타임 게이트 의미론**: A의 CG-002~006(런타임 테스트 통과 요구)은 B의 가장 큰 실질 약점을 정조준 — B는 닫힌 팩의 약 절반(152/322), 특히 최근 119개 연속 팩이 전원 descriptor-only인 채 production_ready로 닫혀 라벨 의미가 희석됨(증빙 레이어는 정직하므로 기만은 아님).
4. **HR/People 스펙**: P14 필러 16개 객체 + rule-engine-first + 민감정보 가드 — B는 HRX 901 unit이 ledger 미반영·RP 미배정.
5. **전략 선언문 품질**: North Star·사용자 10종·비목표 8개·성공기준이 한 문서에 응집 — B에는 사람이 읽는 단일 전략 선언문이 없음(roadmap.md 104줄 골격뿐).
6. (프레임 비판이 추가한 차원) **가독성·인간 감사 가능성**(A 340KB 정독 가능 vs B 258MB JSON 원장), **비용 설계**(A는 위험도별 모델 라우팅 내장, B는 계획 내 비용 차원 전무), **PIPA/노무 컴플라이언스 구체성**, **사용 가능한 제품 도달 경로**(A는 Sprint 5에 Outlook MVP 직진, B는 320팩 후에도 API가 GET 4라우트).

## 4. 판정의 조건부 단서 (프레임 비판 결과)

- **목표 자기참조 순환**: "전략·목표 정합성" 차원이 기준으로 삼은 enterprise SaaS 표준 목표는 Plan B 거버넌스 자신이 생성·유지하는 산물이다. 내부용 vs SaaS는 미결 사업 결정이지 확정 사실이 아니다.
- **사업 결정이 "내부용 우선 + 인간 팀 주도"로 확정되면** 전략 정합성·범위 완전성·조직 적합성 3개 축이 재배열되어 격차가 5:3~4:4까지 좁혀질 수 있다. **그러나 역전은 안 된다** — 검증·거버넌스·적응성의 구조적 우위와, 결정적으로 **전환 비용**(닫힌 팩 320+개·리뷰 영수증 144개·validator 28개·계약 16개가 A 채택 시 전부 고아화, TUW 환산 규칙 부재) 때문이다.
- 평가가 Plan B 리포 안에서 B 친화적 컨텍스트(메모리 지시)와 함께 수행된 환경 편향도 인정함. 이를 보정한 보수적 점수(검증 5.5:7, 분해 5.5:6.5)로도 승자는 동일.

## 5. 결론

**실행 체계(마스터 플랜)로서는 Plan B가 명확히 우월하다 — 7승 1패이고, 보정 후에도 유지된다.** Plan A를 마스터로 채택하는 것은 (a) 착수 가능 단위 0개인 체계로 갈아타면서 (b) 320+팩의 실행 자산을 고아화하고 (c) 소유자의 표준 목표문("closeout-pack plan을 권위적 실행 소스로, 게이트 약화 금지")을 위반하는 선택이라 정당화되지 않는다.

**다만 Plan A는 "더 나쁜 계획"이 아니라 "다른 종류의 문서"다** — 제품 비전·도메인 스펙·기술 결정(특히 M365)·리스크 식별에서는 B가 갖지 못한 실질을 담고 있다. 최적 행동은 양자택일이 아니라:

1. **B를 실행 체계로 유지** (큐·게이트·영수증 그대로)
2. **A를 요구사항·스펙 소스로 흡수** — LDIP/V2 docx 때 쓴 표준 흡수 절차(소스 인덱스 → 요구 추출 → RP 앵커맵 → 갭 adjudication)를 matter 패키지에 동일 적용
3. **A의 우월 요소 4건을 B에 이식**: M365 기술 결정(RP08 계약에 Office.js/NAA/Graph ID 고정), 통합 리스크 레지스터, 런타임 게이트 레이어 분리(descriptor 게이트와 별개 어휘로 — 기존 게이트 약화 없이), HR 스펙의 HRX 901 unit ledger 등재
4. **선결 사업 결정 2건은 사람 몫**: ① 내부용 vs SaaS(또는 "1호 테넌트" 정합화), ② User/Employee 분리 — 둘 다 어느 계획을 따르든 지금 결정해야 소급 비용이 최소.
