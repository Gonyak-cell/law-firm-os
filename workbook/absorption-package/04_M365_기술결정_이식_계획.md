# 04. M365 기술 결정 이식 계획 — no-write 계약 보존

소스: `workbook/matter_dev_docs/08_Microsoft_365_Outlook_Addin_Spec.md` (127줄, SHA256 `05a03499ad30fd47abb508482a04611cbecad8d8d62d9caffb9496feebb7e218`). RP08은 CP00-298로 닫힘 — 모든 부착은 가산·historical-only.

## 0. 핵심 설계 결정 3줄

1. **`contracts/email-dms-core-contract.json`(86키)은 1바이트도 수정하지 않는다.** M365 결정은 신규 **결정 기록(decision-record) 계약** `contracts/email-dms-m365-runtime-contract.json`에 박고, 계약 스스로 `runtime_admission_gate`(전부 false)와 `no_runtime_attestation`으로 "결정은 박되 런타임은 0"을 기계 선언한다.
2. 흡수 절차는 V2(delta) + LDIP(명명객체 전수 체크리스트) 병용. **(검증 반영) 본 워크스트림은 마스터 흡수트랙(문서 02)의 부속 모듈이다**: 산출물은 `docs/matter-pack-integration/m365/` 하위에 `m365-*` 접두로 두고(독립 디렉토리 `docs/matter-docs-integration/`는 폐기), 요구 ID는 `MAT-REQ-M365-###`(마스터 체계 단일화 — `MAT-MISS-*` 사용 금지), 리뷰는 C-MPACK 체계의 부속 관점으로 재배치(아래 §3).
3. **스토리지 결정은 계약 안에서 분리**: 결정 무관 항목(플랫폼·인증·3대 ID·Smart Alerts·배포·필드 스키마)은 즉시 고정, 의존 항목(업로드 타깃 등 5개)은 `blocked_until_storage_decision: true`로 봉인.

## 2. Stage 2 — 신규 계약 + 검증기

### `contracts/email-dms-m365-runtime-contract.json` 구조

```
schema_version: "law-firm-os.email-dms-m365-runtime-contract.v0.1"
status: "decision_record_only"          // descriptor_only와 구분되는 신설 어휘
relationship_to_existing_contracts: { email_dms_core_contract_modified: false,
  no_write_attestation_preserved: true, closed_pack_evidence_immutable: true }
runtime_admission_gate: { planning_gate_complete, storage_decision_resolved: false,
  internal_vs_saas_decision_resolved: false, privilege_classification_decision_resolved: false,
  implementation_allowed: false }       // 핵심 음성 불변식
no_runtime_attestation: { dispatches_filing_runtime: false, calls_graph_api: false,
  acquires_entra_tokens: false, registers_addin_manifest: false,
  executes_smart_alert_handler: false, writes_sharepoint_onedrive: false, ... }
technical_decisions: {
  addin_platform: office_js_outlook_web_addin (COM/VSTO excluded)
  authentication: entra_id + msal + nested_app_authentication_preferred
                  + legacy_exchange_user_identity_tokens: "prohibited"   // 2025 차단 반영
  email_identity: id_triple [graph_message_id, internet_message_id, conversation_id]

filing_flow: F01~F11 (F07 첨부 저장만 storage_dependent)
```

### `scripts/validate-email-dms-m365-runtime-contract.mjs`

1. schema_version·status·필수 키 / 2. 구조 불변식(send_modes 3종, id_triple 3종, F11·A8 연속, 필드 16, 기능 9+8) / 3. **음성 불변식**: `implementation_allowed===false`인 동안 no_runtime_attestation 전 키 false 강제 / 4. **교차 보존 검사(강화 방향)**: 기존 email-dms-core 계약의 descriptor_only===true와 no_write_attestation 18키 false 유지를 assert — **신규 검증기가 기존 no-write 증거의 파수꾼 겸직** / 5. 의존성 봉인: storage_decision_ref===null인 동안 storage_dependent 전 항목 blocked / 6. packages/email-dms import 0건(deepEqual 비결합).

`package.json` 가산 1키: `"rp08:m365-runtime:validate"`.

### rp08 마이크로페이즈 문서 판단

**추가 콘텐츠 불필요** — ① generated 산출물(110 고정)이라 수정 즉사 ② 스키마에 기술 결정 자리가 없음 ③ 닫힌 구간 historical-only. anchor map이 "기술 결정의 권위 소스는 신규 계약"임을 명시.

## 3. 작업 순서·게이트·커밋

순서 (검증 반영 — 마스터 트랙·게이트 레이어 이후 착수): W0 사전 확인(커서·비충돌 grep·MAT-DEC-05) → W1 Stage 1 커밋 → W2 **C-MPACK-M365-01** 리뷰(C-MPACK 체계 부속 관점 — packet에 C-MPACK-02·03과의 스코프 경계와 상호 참조 명기, M365 명명객체의 전수 검사 책임은 본 리뷰가 단독 보유) → W3 Stage 2 커밋 → W4 **C-MPACK-M365-02** 리뷰 → W5(사람 결정 후) admission boolean 갱신 → W6(미래) 승인 확장 팩에서만 `implementation_allowed: true`.

게이트: Stage 1 — SHA256 일치·**74항목 검산**(overlay coverage_summary와 일치 필수)·forbidden 침범 0·리뷰 PASS(p0/p1=0). Stage 2 — 신규 검증기 exit 0·기존 전 스위트 green·리뷰 PASS. 상시 — `email_dms_core_contract_modified===false` 기계 강제, unit delta 0.

커밋: Stage 1 `docs(planning): matter-pack m365 overlay artifacts for rp08 runtime decisions (planning-only)` / Stage 2 `Add email-dms m365 runtime decision contract artifacts (decision-record-only, additive)`. `Close CP00-xxx` 형식은 W6 전 사용 금지.

**수정 금지 목록**: email-dms-core 계약, `packages/email-dms/**`, rp08 마이크로페이즈, `rp-detailed-plan-catalog.mjs`, 닫힌 팩, `validate-closeout-pack.mjs`.
