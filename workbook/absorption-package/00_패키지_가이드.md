패키지 버전: v1.1 (2026-06-11) — 통합 적대적 검증(approve_with_fixes, 게이트 약화 0건) 반영 완료
작성 주체: Claude (17개 독립 분석 트랙 + 조사 3건 + 설계 4건 + 통합 검증 1건 기반)
대상: Law Firm OS 개발팀(Codex 실행 세션 포함), 제품 소유자(jwsuh@amic.kr)

## 1. 이 패키지의 목적

matter 개발문서 패키지(Plan A, `workbook/matter_dev_docs/`)와 리포 RP/CP 체계(Plan B)의 계획 대 계획 비교에서 **Plan B 7승 1패** 판정이 나왔다(`workbook/plan-vs-plan-verdict.md`). 결론은 양자택일이 아니라 **"B를 실행 체계로 유지하고 A의 우월 요소를 흡수"**이며, 이 패키지는 그 흡수를 개발팀이 바로 실행할 수 있는 파일 수준 계획으로 전개한 것이다.

흡수 대상 4건:
1. **HRX 901 unit 등재** — HR을 "차용증"에서 "팩 생성 가능 상태"로 (문서 03)
2. **M365 기술 결정 이식** — Plan A 문서 08의 검증된 결정을 신규 계약으로 (문서 04)
3. **런타임 게이트 레이어** — descriptor/runtime 어휘 분리, Plan A CG-002~006 흡수 (문서 05)
4. **흡수 트랙 + 리스크 레지스터 + 전략 선언문** — Plan A 전체의 표준 흡수 절차 (문서 02)

## 3. 전역 불가침 제약 (모든 워크스트림 공통)

아래는 어느 문서의 어느 단계에서도 위반 불가. 각 계획서에 개별 증명이 있다.

1. **게이트 약화 금지**: plan_totals(54,355 / 901 / 55,256)·boundary 문자열·status enum·`hardenedReviewStartPackNumber=178`·requiredPackFiles 5종 — 전부 기존 값 그대로. 기존 검증기의 검사 **삭제·완화 0건**, 추가만 허용.
2. **닫힌 팩 불변**: `docs/closeout-packs/` manifest 일체 무수정, 리넘버링 금지. 신규 검사는 전부 팩 번호 경계 상수 뒤에서만 발화.
3. **no-write attestation 불변**: `contracts/email-dms-core-contract.json` 등 기존 계약의 false 값은 영구 불변(닫힌 팩 증거). 런타임은 별도 신규 계약으로만.
4. **deepEqual 쌍 불변**: `contracts/*-core-contract.json` ↔ `packages/*/src/registry.js` 쌍, rpXX 마이크로페이즈 JSON(110 entry 고정) 무수정.
5. **LDIP/V2 동결 산출물 무수정**: `docs/ldip-integration/`, `docs/spec-v2-integration/` 참조만.
6. **unit 증가 0 원칙**: 계획 단계에서 `units_currently_added: 0`. 등재는 사용자 승인 ledger 확장(HRX는 문서 03의 승인 절차)으로만.
7. **별도 제품화 금지**: HRX는 내장 모듈(`splits_hrx_product: false` 유지). Plan A 거버넌스 체계(TUW/R0~R14/L0~L13)는 **흡수하지 않고 요구사항만 추출**.
8. **라이브 큐 우선**: 본 패키지의 커서 수치(CP00-323/324, RP10)는 작성 시점 스냅샷이다. **실행 시점에 반드시 `docs/closeout-pack-plan/next-pack-queue.json`과 git log로 재확인**(Stale Context Rule). 진행 중인 CP 큐 작업과의 경로 충돌 금지 — 본 패키지의 신규 경로는 전부 validator 비독해 안전 지대.

## 4. 패키지의 위치와 효력

- 본 패키지는 `workbook/` (CP 거버넌스 밖, 현재 untracked) 소재의 **청사진**이다. 실제 트랙 산출물은 각 계획서가 지정한 경로(`docs/matter-pack-integration/`, `docs/hrx-integration/` 등)에 새로 생성한다.
- 본 패키지 자체는 어떤 구현·ledger·contract·validator·production_ready 선언도 승인하지 않는다.
- 구현 착수 가능 조건: 각 워크스트림의 planning gate 통과 + 해당 MAT-DEC 결정 기록 + 사용자 착수 승인.
