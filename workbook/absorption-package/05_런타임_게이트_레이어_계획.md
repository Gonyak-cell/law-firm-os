# 05. 런타임 게이트 레이어 계획 — descriptor/runtime 어휘 분리

동기: 닫힌 팩의 약 절반이 descriptor-only인 채 production_ready로 닫혀 라벨 의미가 희석됨(기만 아님 — 어휘 부재 문제). Plan A CG-002~006의 런타임 게이트 의미론을 **가산적으로** 흡수한다. (수치 주의: 설계 시점 154/323 → 검증 시점 157/326 — 큐가 하루 ~3팩 속도로 전진하므로 파생 ledger가 동적 재집계하며, 문서 내 고정 수치는 머지 시점 재집계 필수.)

## 0. 핵심 설계 결정 3줄

1. **production_ready의 의미는 1바이트도 안 바꾼다.** status enum {in_progress, production_ready} 불변. `runtime_ready`는 status 값이 아니라 **별도 boolean + 블록**.
2. **`runtime_ready`는 상위 함의 게이트**: true ⇒ production_ready ∧ 기존 전 검사 통과 ∧ RTG-001~005 통과. 런타임 어휘로 production_ready를 우회하는 역방향 경로 없음.
3. **닫힌 ~323팩 무수정.** 분류는 기존 manifest 필드에서 파생하는 신규 ledger가 담당, 신규 검사는 팩 번호 경계 상수 뒤에서만 발화 — `hardenedReviewStartPackNumber=178`과 동일 패턴.

## 5. "게이트 약화 아님" 논증 (지침 준수 증명)

1. **닫힌 팩 불변**: 신규 검사 전부 경계 분기 뒤 — 닫힌 팩 검사 집합 G_pre = G_post. 기계 증명: 전/후 전수 sweep diff 0.
2. **신규 팩 순강화**: 경계 이후 검사 집합 = G_pre ∪ {layer 검사} — 상위집합, 단조 강화.
3. **우회 경로 부재**: runtime_ready는 status가 아니고 production_ready ∧ 기존 전 검사를 함의 — 대체·단축 분기 없음.
4. **재해석 아님, 명시화**: 파생 ledger는 닫힌 manifest가 이미 정직하게 attest한 필드의 순수 함수. `undeclared_legacy` 라벨로 과대 주장도 차단. Plan A CG를 소급 적용하면 닫힌 팩 다수가 미충족이 되는 함정(갭 분석 충돌 4)을 피하는 유일한 가산 경로.
