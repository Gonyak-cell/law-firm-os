# 06. 오픈 결정 레지스터 — 소유자(jwsuh@amic.kr) 결정 필요 항목

본 패키지의 어떤 워크스트림도 **결정을 대신 내리지 않는다.** 각 계획서는 결정 전 가능한 작업과 결정 후 작업을 분리해 두었다. 아래는 마스터 결정 9건 + 워크스트림 로컬 결정의 매핑이다.

**(검증 반영) 본 레지스터가 4개 워크스트림 로컬 결정 ID(HRX D1~D7, M365 ①~⑥, 게이트 D1~D6)의 단일 마스터다.** 동일 결정은 동일 마스터 ID로 통합되며, 데드라인이 복수로 제시된 경우 **가장 이른 시점**이 유효하다. 실행 시 결정 기록은 마스터 트랙 overlay JSON의 `user_decisions_required`에 단일 기록한다.

## 2. 결정의 의존 구조

```
MAT-DEC-02 (User/Employee) ──→ HRX 커밋 2 ──→ HRX 팩 생성 가능
                          └──→ RP11 TimeEntry (라이브 큐 상 가장 먼저 도달)
MAT-DEC-03 (스토리지, 결정됨) ──→ M365 admission에서 storage-decision blocker 해제 ──→ Outlook 런타임 팩은 admin/Graph/runtime evidence 필요
MAT-DEC-01 (내부/SaaS) ──→ M365 배포 분기, 전략 선언문 §6(b), RP19/RP26
MAT-DEC-08 (기밀분류, 결정됨) ──→ M365 privilege 필드, HRX P06 — enum 정책결정 해소, 런타임 evidence 별도
MAT-DEC-05 (소스 동결) ──→ 모든 트랙의 Phase 1 커밋 (가장 먼저 필요한 결정)
MAT-DEC-07 (게이트 레이어) ──→ 문서 05 착수 + M365 계약의 런타임 플래그 명명 (어휘 단일 결정)
```

**결정 없이도 즉시 착수 가능한 작업** (문서 07 참조): 흡수 트랙 Phase 0~4, HRX W1·W2, M365 Stage 1, 게이트 레이어 정책 문서. 즉, **planning 산출물 전체는 결정 대기 없이 생산 가능**하고, 결정은 "커밋 2 / Stage 2 / admission 해제" 지점에서만 관문이 된다.

## 3. 결정별 양 분기 영향 요약

### MAT-DEC-01 (내부 vs SaaS)
- 1호 테넌트 정합화(권고): 기구현 Tenant·cross-tenant deny·RP26/28/29 유지, Plan A 요구는 1호 테넌트 요구사항으로 등재. 손실 없음.
- 내부용 확정: product-contract의 tenant_isolated 원칙·SaaS 하드닝 계약과 충돌 — 계약 개정(게이트 위험)이 필요해져 **비권고**.
- SaaS 확정: Plan A의 M365 admin center 단일 조직 배포 전제 일부 재설계.

### MAT-DEC-02 (User/Employee)
- 분리 채택(권고): HRX·RP11이 모두 깨끗해짐. `Employee.user_id` 역방향이면 닫힌 코어 무변경.
- 미분리: 문서 11 전제 붕괴 — HRX 요구 추출 단계에서 대규모 adapt 필요, Plan A RISK-002(권한 누수) 미완화 지속.

### MAT-DEC-03 (스토리지)
- SharePoint/OneDrive: **2026-06-19 결정됨.** Plan A 문서 08/09 그대로, Graph Drive 표면 해제. 기존 dms `storage_pointer_ref` 추상화가 양쪽을 수용하므로 코어 무변경.
- object storage: M365 filing은 "Graph로 읽고 자체 저장소에 보관" 형으로 filing_flow F07·attachment_flow A03만 재설계.

### MAT-DEC-08 (기밀분류)
- 2026-06-19 결정: `public`, `internal`, `confidential`, `restricted`, `privileged`, `hr_restricted`.
- 기존 4값의 의미는 약화하지 않고, privilege/HR restricted는 더 엄격한 분류로 추가한다.
- 이 결정은 enum/계약 기준을 닫지만, M365/HR 런타임 evidence 또는 실제 데이터 오픈을 승인하지 않는다.

## 4. 기록 방식

결정은 해당 트랙의 overlay JSON `user_decisions_required` boolean 갱신 + adjudication 문서에 결정 일자·결정자·근거를 기록한다. 결정 기록 커밋은 근거 문서 경로를 본문에 동반해야 한다(각 계획서의 커밋 규약 참조).
