## §A 권한 한계와 문서 지위

이 디렉토리의 모든 문서·원장은 **계획이며 어떤 완료 권한도 없다.** 소스오브트루스 위계(라이브 git+manifests > plan JSON > queue > ledger > 증거 > 문서)를 준수하고, 충돌 시 라이브 리포 상태가 우선한다. 구현·원장 변경·계약 변경·production_ready 클레임을 승인하지 않는다. 리포 커밋은 구현 측(Codex) 소관이다.

권위 분리: **구조적 사실(ID, depends_on, vc_bindings, work_type, risk, weight, gate_binding, terminal)은 `launch-tuw-ledger.json`이 단일 소스**이고, **서술(objective, acceptance_criteria 전문, 비고)은 phase MD 파일이 단일 소스**다. 양쪽 ID 집합 일치는 `validate-launch-tuw-ledger.mjs`가 검증한다.

## §C 출시 불변식 (LP1 — 7개)

1. **게이트 적층 비약화**: production_ready ⊂ runtime_ready ⊂ ops_ready ⊂ go_live_approved. 닫힌 팩·기존 게이트 일절 수정 금지(출시 계획 §3).
2. **실데이터 통제**: 실데이터·실크리덴셜 접촉은 production-data-policy 계약 발효(L7 진입) 후, live 단계 goal에 한해서만(출시 계획 §5 L1-4·L7 진입조건).
3. **유닛 등재 단일 경로**: 제품 기능 유닛 등재는 matter-pack overlay admission(사용자 승인 ledger 확장)으로만 — Launch 트랙의 임의 등재 금지(출시 계획 §3 예외 조항).
4. **소스오브트루스 위계 준수**: 본 디렉토리 산출물은 planning-only(§A).
5. **팩 번호 비고정**: 완료 판정은 라이브 원장 + Section 8 게이트 — CP 팩 번호·본 트랙 TUW 번호 모두 재번호 금지, 폐기는 `withdrawn` 상태로만(출시 계획 §2-3).
6. **분담 원칙**: 계획은 Claude, 리포 변경·커밋은 Codex. 인간 결정(executor=human_decision)은 에이전트가 대행 불가.
7. **부분 통과 이월 금지**: 게이트(G1~G10·phase EXIT) 미충족 항목은 P0/P1 finding으로 adjudication 후 게이트 전체 재판정(출시 계획 §5 L8 No-Go 처리).

## §D ID 체계

```
TUW ID:   LT-<PHASE>-W<##>-T<##>     예: LT-L2-W02-T07
WP ID:    LT-<PHASE>-W<##>           예: LT-L2-W02
goal_id:  lt-<phase>-w<##>           예: lt-l2-w02  (소문자-하이픈 — 기존 goal-closeout 명명과 동형)
PHASE ∈ { PRE, L0, L1, L2, L3, L4, L5, L6, L7, L8, L9 }
```

- **위치 정보만 인코딩**: 의미(도메인·유형·리스크)는 전부 필드로. 기존 `MAT-…-TUW-###` 체계(문서 17)는 레지스트리 부재·자기형식 위반·보일러플레이트로 "착수 가능 인스턴스 0개" 판정(plan-vs-plan-verdict.md:28)을 받은 전례가 있어 승계하지 않는다. 문서 17·18은 수정하지 않고 레거시로 보존한다.
- **정렬 권위**: ID 문자열 사전순이 아니라 원장 `meta.phase_order` 배열이 실행 순서의 권위.
- **재번호 금지**: 발번된 ID는 재사용·재번호 불가. 폐기는 `status: withdrawn`.
- **별칭(alias)**: 출시 계획 원문의 goal 명명(GL-MIG-01 등)은 WP 레지스트리 alias 열에 보존(추적성).
- **terminal 메커니즘**: 각 WP는 정확히 1개 TUW에 `terminal: true`를 표기(통상 WP 완료 게이트 검증·증거 조립 단위). 교차 WP 의존 `WP:LT-L2-W02` 표기는 그 WP의 terminal TUW로 해석된다. 레지스트리의 terminal_tuw 열은 병합 시 자동 기입.

|---|---|---|---|

| 5 | `deliverables` | 배열 | 생성·수정 파일 경로 또는 비코드 산출물(결정 기록, consent 증적 등) |
| 6 | `source_refs` | 배열 | 출시 계획 원문 절 앵커 + 관련 계약·문서·코드 경로("읽을 파일" 겸함) |

| 8 | `external_dependencies` | 키 배열 | §F 외부 의존 레지스트리 키만 |
| 9 | `executor` | enum | `human_decision \| agent_implementation \| external_party \| hybrid` |
| 10 | `risk_class` | enum | `A \| B \| C` — sensitivity-first. 자동-A 키워드(§H-6) 검출 시 A 강제 |
| 11 | `weight` | enum | `L(0.5~2h) \| M(2~6h) \| H(1~3d)` — XH 금지(분해 의무) |
| 12 | `acceptance_criteria` | 배열 | **검증 가능 문장만**: 각 항목이 측정값·exit code·존재 판정 중 하나로 환원. 동일 문구 반복 기계검출 |
| 13 | `verification_contract` | 객체 | `{ method, evidence, vc_bindings[] }` — method는 실행 커맨드 또는 번호 붙은 수동검증 절차(일반 문구 금지) |

| 16 | `constraints_overrides` | 문자열(조건부) | 공통 프로토콜 §0(§F)에서 벗어나는 것만 기록 |

## §F 공통 프로토콜 §0 (전 TUW 상속 — 개별 복제 금지)

ABS 백로그(workbook/absorption-package/08_Codex_TUW_백로그.md) 선례 승계. 모든 TUW는 아래를 자동 상속하며, 벗어나는 경우만 `constraints_overrides`에 기록한다.

1. **전역 금지 경계**: `contracts/` 기존 파일, `packages/`, `scripts/`, `docs/closeout-packs/`, `docs/goal-closeout/` 기존 폴더, 닫힌 팩 증거 일체 수정 금지. 신규 파일은 해당 TUW deliverables에 명시된 경로만 생성.
2. **Stop Condition**: 사양 충돌·모호성·불변식(§C) 저촉 가능성 발견 시 작업 중단 → BLOCK 기록 → 에스컬레이션. 추측 강행 금지.
3. **Escalation**: 인간 결정 필요 사안은 launch-decision-register(L1 산출물)로 회부. executor=human_decision TUW를 에이전트가 대행 완료 처리 금지.
4. **Loop Budget**: TUW당 구현-검증 시도 3회 한도. 초과 시 BLOCK 기록 후 WP 오너 판단.
5. **Model Routing**: risk_class=A는 상위 모델 + read-only 교차 리뷰 의무(CP 승계 규칙).
6. **완료 의식**: verification_contract.method 실행 → 출력(exit code·카운트)을 work-log에 기록 → WP의 command-evidence.json에 TUW ID로 귀속. "커밋이 증거가 아니라 마지막 단계"(goal-closeout 원칙).
7. **실데이터 금지(기본값)**: §C-2 발효 전 모든 TUW는 합성 데이터·샌드박스 한정.

## §J Phase 간 의존 개요 (WP-level DAG)

```
PRE(전건) ──────────────────────────────────────────┐
L0(W01~05) → L1(W01~10) → ┬ L2(W01~07; W02가 첫 goal) ─┬→ L5(W01~07) ─┬→ L7(W01~06) → L8(W01~03) → L9(W01~05)
                           ├ L3(W01~10; L2와 병렬)      ─┤  L6(W01~06)  ┘
                           └ L4(W01~06; L2 부분 가동 후) ─┘  (L6은 L3-W06 이후 L4·L5와 병렬)
```

- 의존 방향 규칙: TUW는 같은 phase 또는 phase_order상 선행 phase에만 의존 가능(검증기 강제). 병렬 트랙(L2·L3·L4, L5·L6)은 phase_order상 순번이 있으므로 역방향 의존이 필요하면 작업을 앞 phase로 옮긴다.
- L8-W02의 각 게이트 조립 TUW는 G1~G10 증거 생산 WP의 terminal에 의존 — 게이트 도달 경로의 종결점.
